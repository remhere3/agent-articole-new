Pornire rapida
  1. Creeaza mediu virtual si instaleaza dependentele:
    python3 -m venv venv && source venv/bin/activate && pip install -r requirements.txt
  2. Copiaza si completeaza configuratia:
    cp .env.example .env — pune cel putin ANTHROPIC_API_KEY sau TAVILY_API_KEY
  3. Porneste serverul:
    python run.py
  4. Deschide browser la http://localhost:8007
  5. Adauga un utilizator → creeaza un topic → apasa pentru prima cautare

Web search nativ vs. Tavily

Singurul provider cu web search nativ este anthropic — Claude foloseste tool-ul web_search_20250305 direct, fara nicio cheie externa suplimentara.

Modelele Ollama locale (llama3, gemma4:e4b, mistral etc.) nu au acces la internet — ele ruleaza 100% local si nu pot cauta nimic pe cont propriu. Providerul ollama din aceasta aplicatie foloseste Tavily pentru cautarea web si trimite rezultatele catre modelul local doar pentru rezumare si structurare. Prin urmare, TAVILY_API_KEY este obligatorie si pentru ollama.

Cum maximizezi numarul de rezultate
Limita structurala: ~10 articole per run (provider Anthropic)

Tool-ul web_search_20250305 permite maxim 5 apeluri de cautare per run. Fiecare apel returneaza 2–4 articole relevante dupa filtrare de data. Rezultatul tipic este 5–15 articole per run, indiferent de complexitatea intrebarii.

Aceasta nu e o limitare a aplicatiei — e o limita tehnica a tool-ului nativ Anthropic. Providerul tavily poate returna mai multe rezultate (20–30+) per run deoarece face 4 cereri paralele (2 interogari × 2 treceri academic/general).

Cum formulezi user_question optim

Ce face user_question: ghideaza subiectul cautarii cu mai mult context decat keywords-urile singure. Claude il foloseste ca referinta de domeniu, nu ca instructiune de formatare a raspunsului.
RegulaCorectGresit — cauzeaza 0 rezultate
1–2 propoziții max, limbaj de căutare Recent papers on green hydrogen production costs, PEM/SOEC electrolysis efficiency and LCOH projections 2030-2050 Prompt de 1000+ cuvinte cu 7 secțiuni și tabele cerute
Fara instructiuni de format Find peer-reviewed articles on hydrogen storage ammonia LOHC carriers published after 2023 Present results as: (a) structured summary, (b) table, (c) 10 must-read papers with DOI
Fara instructiuni de rol AI Hydrogen production industry: SMR, electrolysis, CCS lifecycle emissions You are an expert research assistant specializing in...
Poti lasa gol dacă keywords sunt suficienți (camp gol) + keywords: LCOH electrolyzer stack cost 2030 Intrebare complexa fara keywords de backup

Setarea days_back — impact major

Peer-review dureaza 2–6 luni. Cu days_back=30 pe un subiect nis, exista fizic putine articole noi — nu e o problema a agentului.

Strategia cu topicuri separate — pentru review-uri comprehensive

Daca vrei sa acoperi un domeniu larg (ex: industria hidrogenului cu 7 dimensiuni), creeaza un topic separat per dimensiune in loc de un singur topic cu o intrebare complexa. Fiecare run va returna 5–15 articole focalizate. Dupa 7 run-uri ai 35–100 articole categorializate.

De ce 2 rulari succesive nu returneaza aceleasi rezultate?

Acest comportament este normal si intentionat. Exista trei cauze:

  1. Cautarile web sunt live si non-deterministe — fiecare run face cereri reale catre motoare de cautare. Rankingul se schimba, articole noi apar, cache-ul difera de la un apel la altul.
  2. Modelul LLM are variabilitate — Claude decide singur ce interogari sa lanseze si ce articole sa includa. Cu temperature > 0 (implicit), aceleasi instructiuni pot genera strategii de cautare usor diferite la fiecare rulare.
  3. Filtrarea dupa data taie diferit la margine — articolele publicate exact la limita ferestrei days_back pot fi incluse sau excluse in functie de ora rularii sau de cum modelul interpreteaza data.

Recomandare: foloseste days_back mai mare (30–60 zile) pentru a mentine articolele importante in fereastra pe mai multe rulari. Rularea zilnica (nu la minute distanta) este comportamentul optim — prinde articolele noi fara zgomot.

Functionalitati avansate
#ImbunatatireImpact
1 Rezultate complete per run — fiecare rulare salveaza toate articolele gasite; filtrare exclusiv dupa days_back Nu se mai pierd articole din cauza deduplicarii cross-run; poti compara rezultatele intre rulari
2 Timeout per topic — camp timeout_seconds (default 300s) pe Topic, wrapper asyncio.wait_for() in orchestrator Cautarile blocate nu ocupa scheduler-ul la infinit; run-ul e marcat error cu mesaj clar
3 Validare API key — GET /api/searches/validate-provider/{provider} Poti verifica daca cheia API functioneaza inainte de a o folosi intr-o cautare reala
4 Protectie prompt injection — user_question invelit in taguri XML <user_research_topic> in prompt Utilizatorii maliciosi nu pot injecta instructiuni in Claude prin campul de intrebare
5 Migrare schema fara Alembic — _ensure_columns() verifica si adauga coloane noi via PRAGMA table_info + ALTER TABLE Migrare idempotenta la pornire, fara a reseta configuratia de logging (problema cauzata de Alembic's fileConfig)
6 Preview email — GET /api/searches/runs/{run_id}/preview-email Poti vedea in browser cum arata raportul email al unui run, fara sa-l trimiti efectiv
7 Cost estimat per rulare — camp estimated_cost_usd pe SearchRun Vizibil in UI si in email; ajuta la monitorizarea costurilor Anthropic in timp
8 Rate limiting trigger manual — cooldown 60s pe POST /api/searches/run/{topic_id} Previne declansarea accidentala a mai multor cautari consecutive pe acelasi topic
9 Tab Articole in UI — vizualizare flat a tuturor articolelor cu search client-side, toggle deduplicare URL, export CSV/JSON Nu mai trebuie sa navighezi prin fiecare run separat; poti cauta si exporta global
10 Export articole — GET /api/searches/results/export?format=csv|json Export complet fara limita de paginare, filtrat optional pe topic; util pentru analize externe
11 Robustete prompt Anthropic — prompt simplificat (fara sectiuni all-caps care declansau reasoning-ul), traducere automata a topicului in engleza, fallback synthesis call daca modelul nu genereaza text final, fallback reformatting call daca returneaza proza in loc de JSON Cautarile cu topicuri in romana sau cu subiecte niche returneaza articole in loc de []
Migrare DB automata — La fiecare pornire, _ensure_columns() verifica via PRAGMA table_info daca coloanele noi exista (timeout_seconds, estimated_cost_usd, tokens_input, tokens_output, api_calls) si le adauga daca lipsesc. Operatia e idempotenta si nu modifica date existente. Nu e nevoie de Alembic sau de actiuni manuale.
Log-uri SSE intotdeauna active — Stream-ul SSE porneste automat la incarcarea paginii (nu doar la deschiderea panoului de log-uri). Apasa butonul FAB terminal din stanga-jos pentru a deschide/inchide panoul — inchiderea nu opreste stream-ul. Badge-ul rosu arata numarul de mesaje noi necitite.

import httpx

BASE = "http://localhost:8007/api"
c = httpx.Client(timeout=120)

def api(method, path, **kw):
    r = c.request(method, BASE + path, **kw)
    r.raise_for_status()
    return r.json()

# 1. Configurare
api("PUT", "/settings/anthropic_api_key",
    json={"key":"anthropic_api_key","value":"sk-ant-..."})

# 2. Utilizatori
u1 = api("POST", "/users", json={"name":"Ana","email":"ana@research.ro"})
u2 = api("POST", "/users", json={"name":"Mihai","email":"mihai@clinic.ro"})

# 3. Topicuri
topics = [
    {
        "name": "CRISPR 2026",
        "user_question": "Articole recente despre CRISPR si editare genomica",
        "keywords": "CRISPR gene editing genomics",
        "days_back": 7,
        "provider": "anthropic",
        "user_ids": [u1["id"], u2["id"]]
    },
    {
        "name": "AI Climate",
        "keywords": "machine learning climate change models",
        "days_back": 14,
        "provider": "tavily",
        "user_ids": [u1["id"]]
    },
]
for cfg in topics:
    t = api("POST", "/topics",
            json={**cfg, "periodicity_hours": 24, "send_email": True, "active": True})
    print(f"Topic #{t['id']}: {t['name']} ({t['provider']})")

# 4. Ruleaza toate
for t in api("GET", "/topics"):
    if t["active"]:
        run = api("POST", f"/searches/run/{t['id']}")
        print(f"  {t['name']}: {run['status']} — {run['results_count']} articole")
        for a in run["results"][:3]:
            print(f"    [{a['published_date']}] {a['title'][:60]}")

c.close()

Notificari ntfy

ntfy.sh este un serviciu open-source de notificari push prin HTTP. Dupa fiecare rulare (success sau error), agentul trimite o notificare cu rezultatele pe telefonul sau browserul tau.

  1. Instaleaza aplicatia ntfy pe telefon (Android / iOS)
  2. Alege un nume de topic unic (ex: agent-articole-abc123) — oricine cunoaste numele poate primi notificarile, deci alege ceva greu de ghicit
  3. Aboneaza-te la topic in aplicatie
  4. Seteaza variabilele in .env si reporneste serverul

Exemplu notificare (success):

✅ Agent Articole — industria hidrogenului 8 articole găsite în 42.3s Abonați: Mihai, Ana Telemetrie: req: 5 · in: 12,450 · out: 1,823 Provider: anthropic · Run #36

Exemplu notificare (error):

❌ Agent Articole — industria hidrogenului 0 articole găsite în 3.1s Abonați: Mihai Provider: anthropic · Run #37
Variabila .envDefaultDescriere
NTFY_ENABLED false Activeaza/dezactiveaza notificarile (true / false). Comuta rapid fara sa stergi celelalte variabile.
NTFY_TOPIC — Numele topicului, ex: agent-articole-abc123. Alege ceva greu de ghicit — oricine il cunoaste poate primi notificarile.
NTFY_BASE_URL https://ntfy.sh Serverul ntfy. https://ntfy.sh pentru cloud sau http://localhost:8080 pentru instanta locala Docker.
ntfy local (Docker) — inclus in docker-compose.yml
Proiectul include un docker-compose.yml care porneste ntfy pe portul 8080:
docker compose up -d
Apoi in .env:
NTFY_ENABLED=true
NTFY_BASE_URL=http://localhost:8080
NTFY_TOPIC=agent-articole
Aboneaza-te in aplicatia ntfy la http://<ip-masina>:8080 → topic agent-articole.

Deployment ca serviciu systemd

Pentru a rula agentul permanent pe un server Linux, creeaza un serviciu systemd:

sudo nano /etc/systemd/system/agent-articole.service

Continut fisier:

[Unit]
Description=Agent Articole FastAPI
After=network.target

[Service]
Type=exec
User=ubuntu
WorkingDirectory=/home/ubuntu/claude/agent_articole
ExecStart=/home/ubuntu/claude/agent_articole/venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8007 --log-level info
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target

Activeaza si porneste serviciul:

sudo systemctl daemon-reload
sudo systemctl start agent-articole
sudo systemctl enable agent-articole
sudo systemctl status agent-articole

Comenzi utile:

sudo journalctl -u agent-articole -f       # log-uri live
sudo systemctl restart agent-articole      # repornire dupa modificari de cod
sudo systemctl stop agent-articole         # oprire
De ce un singur worker?
Aceasta aplicatie foloseste APScheduler (pornit in lifespan) si SSE log streaming (stare in memorie). Cu mai multi workers, fiecare process porneste propriul scheduler — joburile s-ar executa de N ori simultan. In plus, log stream-ul din UI ar afisa doar log-urile unui singur worker.

Un singur process async (FastAPI + AsyncIOScheduler) este suficient — concurenta este gestionata de event loop, nu de procese multiple. Nu adauga --workers N.
